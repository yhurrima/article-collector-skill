# AGENTS.md

## 核心规则

### 1. 方案预审
- 编写或修改代码前，先描述方案并等待用户确认。
- 如果需求不明，先提出澄清问题。

### 2. 任务拆解
- 如果一项任务涉及修改超过 3 个文件，先停下并拆解为更小任务。

### 3. 测试驱动修复
- 发现 bug 时，先写能复现该 bug 的测试，再修复直到测试通过。
- 修复完成后至少运行相关单测；涉及核心流程时运行完整测试。

### 4. API 摘要硬规则
- 文章摘要必须调用模型 API。
- 如果 API 不可用、模型返回错误或鉴权失败，必须报错或写入 `处理状态=处理失败`。
- 禁止用网页文本拼接假摘要作为兜底。
- 用户输入 API 配置时，优先询问通用 `BASE_URL`、`AUTH_TOKEN`、`AI_MODEL`，不要只限定 `ANTHROPIC_*` 命名。
- 不要把用户 token 硬编码进仓库或文档。

### 5. 飞书字段约定
- 当前只使用一个 `主要内容` 字段。
- 不再写入 `主要内容-简短`、`主要内容-中等`、`主要内容-详细`。
- `summaryLength` 决定写入 `主要内容` 的篇幅。
- 失败文章应保留链接，并写 `处理状态=处理失败`。

### 6. 抓取与代理
- 默认抓取失败后，应关闭系统代理再重试。
- 页面正文提取应优先使用正文容器，并清理作者简介、订阅入口、相关阅读、广告、评论等噪音。
- 对仍无法抓取的网站，保留失败状态，不要伪造正文或摘要。

### 7. 飞书文档格式
- `主要内容` 如果已经包含编号，生成飞书文档时不要再嵌套有序列表。
- 防止出现 `1.` 空项、`a.` 子项、二次编号等格式问题。

### 8. 首次使用流程
- 新用户或 `onboardingComplete=false` 时，必须先询问:
  - 发送报告时间
  - 阅读汇总篇幅
  - API 配置
- 必须强烈推荐安装浏览器插件，并询问用户是否安装。
- 同时说明无插件方案: 用户可直接说“收藏这篇文章，这是文章链接：<url>”。

### 9. 文档同步
- 完成涉及可复用经验、运行方式、配置方式、坑点的任务后，主动询问是否更新 `README.md`、`memory.md`、`AGENTS.md`。
- `README.md` 记录运行方式、命令、依赖、目录结构、使用步骤、验证方式。
- `memory.md` 记录长期经验、关键决策、原因、影响、后续注意。
- `AGENTS.md` 记录项目级工作规则和约束。
